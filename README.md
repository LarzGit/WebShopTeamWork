# MegaMart — веб-магазин на Django

Командний навчальний проєкт: повноцінний вебмагазин, де відвідувач переглядає товари,
фільтрує їх, кладе в кошик і оформлює замовлення, а адміністратор керує каталогом,
замовленнями й користувачами.

## Технології

- **Backend:** Django 5.2 + Django REST Framework
- **Frontend:** серверний рендеринг на Django-шаблонах (Bootstrap 5) — застосунок `shop`
- **БД:** SQLite (за замовчуванням, для розробки)
- **Автентифікація:** сесії Django (cookie-based), кастомна модель користувача з ролями

## Архітектура (застосунки)

| Застосунок | Відповідає за |
|---|---|
| `accounts` | Кастомна модель користувача (`CustomUser` з полями `phone`, `role`), реєстрація/вхід/вихід/профіль через REST API, дозволи (`permissions.py`) |
| `catalog` | Товари, категорії, відгуки — моделі + REST API (`/api/catalog/...`) з пошуком, фільтрами, сортуванням, пагінацією |
| `orders` | Кошик, позиції кошика, замовлення, позиції замовлення — моделі + REST API (`/api/orders/...`) |
| `shop` | Публічний веб-фронтенд (Django-шаблони): головна, каталог, картка товару, кошик, оформлення замовлення, профіль, історія замовлень |
| `projekt` | Налаштування проєкту, кореневий `urls.py` |

Django-адмінка (`/admin/`) використовується адміністратором для керування товарами,
категоріями, замовленнями (зі зміною статусу) та користувачами (блокування через `is_active`).

### Ролі користувачів

- **Гість** — перегляд каталогу, пошук, фільтри, картка товару.
- **Покупець** (`role=customer`) — усе, що гість, плюс кошик, оформлення замовлення,
  історія замовлень, редагування профілю, відгуки.
- **Адміністратор** (`role=admin`, зазвичай ще й `is_staff=True`) — керування каталогом
  (створення/редагування/видалення товарів і категорій — обмежено дозволом
  `IsAdminOrReadOnly`), зміна статусу будь-якого замовлення, керування користувачами
  через `/admin/`.

## Запуск проєкту "з нуля"

```bash
# 1. Клонувати репозиторій і перейти в теку проєкту
git clone <адреса репозиторію>
cd WebShopTeamWork-backBranch

# 2. Створити та активувати віртуальне середовище
python -m venv .venv
# Windows:
.venv\Scripts\activate
# Linux / macOS:
source .venv/bin/activate

# 3. Встановити залежності
pip install -r requirements.txt

# 4. Застосувати міграції бази даних
python manage.py migrate

# 5. (опційно) Наповнити базу тестовими даними
python manage.py seed_data

# 6. Створити суперкористувача для доступу до /admin/
python manage.py createsuperuser

# 7. Запустити сервер розробки
python manage.py runserver
```

Після цього сайт доступний на `http://127.0.0.1:8000/`, а адмінка — на
`http://127.0.0.1:8000/admin/`.

> Щоб користувач мав права адміністратора магазину (а не лише доступ до `/admin/`),
> встановіть йому `role = "admin"` — або через `/admin/`, або в консолі Django
> (`python manage.py shell`).

## Основні маршрути

### Веб-інтерфейс (`shop`)
| URL | Опис |
|---|---|
| `/` | Головна |
| `/catalog/` | Каталог з пошуком/фільтрами/сортуванням |
| `/product/<id>/` | Картка товару + відгуки |
| `/register/`, `/login/`, `/logout/` | Автентифікація |
| `/profile/` | Профіль користувача |
| `/cart/` | Кошик |
| `/checkout/` | Оформлення замовлення |
| `/orders/`, `/orders/<id>/` | Історія та деталі замовлень |

### REST API
| URL | Опис |
|---|---|
| `GET /api/catalog/products/` | Список товарів (пошук/фільтри/сортування/пагінація) |
| `GET /api/catalog/products/<id>/` | Деталі товару |
| `POST/PUT/DELETE /api/catalog/products/...` | Керування товарами — **тільки роль admin** |
| `GET/POST /api/catalog/categories/` | Категорії — читання всім, запис — **тільки admin** |
| `GET/POST /api/catalog/reviews/` | Відгуки — запис доступний будь-якому автентифікованому користувачу (тільки автор може редагувати/видаляти свій відгук) |
| `POST /api/auth/register/`, `/login/`, `/logout/` | Автентифікація |
| `GET/PATCH /api/auth/profile/` | Профіль поточного користувача |
| `GET /api/orders/cart/`, `POST .../cart/add/`, `PATCH .../cart/update/<id>/`, `DELETE .../cart/remove/<id>/` | Кошик |
| `POST /api/orders/checkout/` | Оформлення замовлення |
| `GET /api/orders/`, `/api/orders/<id>/` | Історія та деталі замовлень |
| `PATCH /api/orders/<id>/status/` | Зміна статусу замовлення — **тільки роль admin** |

## Тестові сценарії (ручне тестування)

1. Реєстрація нового користувача → вхід → редагування профілю.
2. Пошук товару за словом у назві/описі; комбінація фільтрів (категорія + ціна + рейтинг)
   разом із сортуванням.
3. Додавання товару в кошик двічі — кількість в одній позиції збільшується, а не дублюється.
4. Спроба покласти в кошик більше, ніж є на складі — має бути відмова.
5. Оформлення замовлення → кошик порожніє → замовлення з'являється в історії з правильною
   зафіксованою сумою.
6. Вхід адміністратором у `/admin/` → зміна статусу замовлення → перевірка, що новий статус
   бачить сам користувач у своїй історії замовлень.
7. Блокування користувача (`is_active = False` в `/admin/`) → перевірка, що він не може
   увійти ні через веб-форму, ні через `/api/auth/login/`.

## Відомі обмеження / що не входить у MVP

- Кошик для гостя (без реєстрації) не реалізований — кошик прив'язаний до
  автентифікованого користувача та зберігається в базі даних.
- Список бажань (wishlist), промокоди/знижки та email-сповіщення не реалізовані —
  це необов'язкові розширення поза MVP.
- Оплата — імітація (вибір способу оплати без реальної інтеграції з платіжною системою).
